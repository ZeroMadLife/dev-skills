# Skills 有效性评估标准

本文档定义如何评估 dev-skills 中的 skills 是否真正有效，以及何时应该保留、优化或移除某个 skill。

---

## 评估维度

采用 **A/B/C/D 四维度结合评估**：

### A. 使用频率（Usage Frequency）

**方法**：记录每个 skill 的调用次数和使用场景。

**评估周期**：每周复盘一次。

**记录方式**：
```markdown
## 本周使用记录（2026-06-13 ~ 2026-06-20）

- logic-lens: 3 次
  - 场景1：OAuth2 重试逻辑重构前审查
  - 场景2：接入侧配置保存代码审查
  - 场景3：面试前优化代码示例
  
- user-thoughts: 5 次
  - 记录 OAuth2 重试决策
  - 记录分布式锁权衡
  - 记录字段映射设计原因
  - 记录被拒绝的 Caffeine 方案
  - 更新 plans.md 开发进度

- prompt-intensity: 自动触发，统计无意义
```

**触发动作**：
- **0 次使用** → 考虑移除（可能是场景不匹配或触发条件设计不当）
- **1-2 次/周** → 保留观察
- **3+ 次/周** → 高价值，继续使用

---

### B. 时间节省（Time Saved）

**方法**：对比"有 skill"和"没有 skill"的时间差。

**评估周期**：每次使用时主观估算。

**记录方式**：
```markdown
## 时间节省记录

| 日期 | Skill | 场景 | 有 skill | 无 skill | 节省时间 |
|------|-------|------|---------|---------|---------|
| 2026-06-13 | logic-lens | OAuth2 重试审查 | 10 min | 30 min | 20 min |
| 2026-06-14 | user-thoughts | 记录架构决策 | 2 min | 15 min（找回上下文） | 13 min |
| 2026-06-15 | prompt-intensity | 自动判断强度 | 0 min | 2 min（手动判断） | 2 min |
```

**触发动作**：
- **节省 < 5 min** → 重新评估该 skill 的价值
- **节省 5-15 min** → 中等价值
- **节省 > 15 min** → 高价值

**特殊情况**：user-thoughts 的真正价值在"新会话恢复上下文"，不在"写入时的时间"。如果某次新会话因为读了 `.ustht/` 省了 30 分钟重新理解项目，也应该记录。

---

### C. 输出质量（Output Quality）

**方法**：主观评分，每次使用后打分（高/中/低价值）。

**评估周期**：每次使用时即时评分。

**评分标准**：
- **高价值**：发现了我没注意到的关键问题，或显著提升了输出质量
  - 例：logic-lens 发现竞态条件，避免生产事故
  - 例：user-thoughts 让新会话立即恢复上下文，无需重复解释

- **中价值**：有帮助，但不是必不可少
  - 例：logic-lens 发现了些代码风格问题，但不影响功能
  - 例：prompt-intensity 判断对了强度，但我本来也能判断

- **低价值**：输出质量一般，或没有实质性帮助
  - 例：logic-lens 只发现了些无关紧要的小问题
  - 例：user-thoughts 写的决策太浅显，没有提供新视角

**触发动作**：
- **连续 3 次低价值** → 考虑移除或优化 skill
- **高价值占比 > 50%** → 高价值 skill，继续使用

---

### D. Prompt 强度自动判断（Automated Intensity-Based Triggering）

**方法**：在 prompt 到达时自动评估复杂度，决定触发哪些 skills。

**评估周期**：实时（每次会话开始时）。

**判断规则**：见 `prompt-intensity/SKILL.md`。

| 强度 | 场景 | 触发 skill | 读取记忆 |
|------|------|-----------|---------|
| 轻度 | 查询、小修改、看日志 | 无 | 不读 |
| 中度 | 重构、新功能、单模块 | logic-lens | rules + plans |
| 重度 | 架构设计、多模块、技术选型 | brainstorming + user-thoughts + logic-lens | 全部维度 |

**触发动作**：
- 判断准确率 < 80% → 优化 prompt-intensity 的判断逻辑
- 判断准确率 > 90% → 当前逻辑有效

**如何评估准确率**：
```markdown
## Prompt 强度判断准确率（本周）

| 日期 | 实际任务 | 自动判断 | 人工判断 | 是否准确 |
|------|---------|---------|---------|---------|
| 2026-06-13 | 重构 OAuth2 | 中度 | 中度 | ✅ |
| 2026-06-14 | 设计认证架构 | 重度 | 重度 | ✅ |
| 2026-06-15 | 看 git log | 轻度 | 轻度 | ✅ |
| 2026-06-16 | 加日志查询接口 | 中度 | 重度 | ❌（低估） |

准确率：75% (3/4)
```

---

## 综合评估流程

### 每周复盘（周五下午）

1. 统计本周各 skill 的使用频率（维度 A）
2. 汇总本周的时间节省记录（维度 B）
3. 汇总本周的输出质量评分（维度 C）
4. 计算 prompt-intensity 的判断准确率（维度 D）

### 决策矩阵

| 使用频率 | 时间节省 | 输出质量 | 决策 |
|---------|---------|---------|------|
| 0 次 | - | - | **移除** |
| 1-2 次 | < 5 min | 低价值 | **观察 1 周，仍低则移除** |
| 1-2 次 | > 5 min | 中/高价值 | **保留** |
| 3+ 次 | > 5 min | 高价值 | **高价值 skill，推荐给团队** |

---

## 优化方向

根据评估结果，skill 可以有三种处理方式：

### 1. 保留（Keep）
- 高频使用 + 高时间节省 + 高输出质量
- 无需调整

### 2. 优化（Optimize）
- 中频使用，但时间节省或输出质量不足
- 优化方向：
  - 调整触发条件（prompt-intensity 规则）
  - 优化 skill 内部逻辑
  - 补充示例或使用指南

### 3. 移除（Remove）
- 0 次使用或连续低价值
- 移除前确认：
  - 是不是场景不匹配（skill 本身没问题，只是你的项目用不上）
  - 是不是触发条件设计不当（skill 有价值，但没被触发）

---

## 示例：logic-lens 评估

**第 1 周评估结果**：
- 使用频率：3 次
- 时间节省：平均 20 min/次
- 输出质量：2 次高价值（发现竞态条件、空指针）、1 次中价值
- 决策：**保留，高价值 skill**

**第 2 周评估结果**：
- 使用频率：0 次
- 原因分析：本周没有重构任务，只做了小修改
- 决策：**保留观察，场景不匹配不代表 skill 无效**

**第 3 周评估结果**：
- 使用频率：5 次
- 时间节省：平均 15 min/次
- 输出质量：4 次高价值、1 次中价值
- 决策：**保留，推荐给团队**

```markdown
# Skills 评估周报

## 使用频率（维度 A）

| Skill | 使用次数 | 场景 |
|-------|---------|------|
| logic-lens | 3 | OAuth2 审查、配置保存审查、面试代码优化 |
| user-thoughts | 5 | 架构决策记录 |
| prompt-intensity | 自动触发 | - |

## 时间节省（维度 B）

| Skill | 平均节省时间 | 备注 |
|-------|------------|------|
| logic-lens | 20 min | 发现了竞态条件，避免重写 |
| user-thoughts | 10 min | 新会话恢复上下文 |

## 输出质量（维度 C）

| Skill | 高价值 | 中价值 | 低价值 |
|-------|--------|--------|--------|
| logic-lens | 2 | 1 | 0 |
| user-thoughts | 3 | 2 | 0 |

## Prompt 强度判断准确率（维度 D）

- 总判断次数：10
- 准确次数：9
- 准确率：90%

## 本周决策

- logic-lens：保留，高价值
- user-thoughts：保留，高价值
- prompt-intensity：准确率 90%，有效
```
